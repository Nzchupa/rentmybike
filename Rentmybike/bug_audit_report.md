# RentMyBike — Повний аудит коду (баги та помилки)

Перевірено весь backend (Spring Boot, 58 Java-файлів + SQL-міграції + конфіги) та весь frontend (Next.js/TypeScript, всі .ts/.tsx файли).

## CRITICAL

| # | Місце | Проблема |
|---|---|---|
| 1 | `application.yml` / `application-prod.yml` | Prod-профіль не перевизначає `app.jwt.secret`. Якщо env-змінна `JWT_SECRET` не задана на сервері — застосунок тихо використовує дефолтний секрет із репозиторію → будь-хто може підробити JWT. |
| 2 | `BikeCategory.java` vs `V1__init_schema.sql` | Java enum містить `HYBRID`, якого немає в Postgres ENUM `bike_category`. Створення/оновлення велосипеда з цією категорією падає з 500. |
| 3 | `BookingService.createBooking()` | Race condition: перевірка перетину дат і вставка бронювання — окремі запити без блокування рядків і без БД-обмеження (`EXCLUDE`/unique). Два orendarі можуть одночасно забронювати один велосипед на ті самі дати (double-booking). |
| 4 | Frontend: завантаження фото велосипеда | Сторінки `/dashboard/bikes/[id]/photos` не існує взагалі, хоча API (`uploadPhoto`/`deletePhoto`) реалізовано. Функція повністю недоступна користувачу — кнопка веде на 404. |

## HIGH

**Backend**
- `AuthService.logout()` / `refresh()` — немає server-side відкликання refresh-токена; вкрадений токен лишається дійсним до 7 днів навіть після "виходу".
- `JwtAuthenticationFilter` — перевіряє лише `isEnabled()`, не `isAccountNonLocked()`. Забанений користувач продовжує працювати з активним токеном до його природного закінчення.
- `BookingService.acceptBooking()` — не перевіряє повторно конфлікт дат при підтвердженні (підсилює проблему #3).

**Frontend**
- `lib/api.ts` — `refreshToken` визначено, але ніде не викликається; немає interceptor-а на 401 → сесія "зависає" як залогинена, хоча токен вже невалідний.
- `middleware.ts` — не виконує жодної авторизації/автентифікації, лише locale-routing. Захист `/dashboard` і `/admin` — тільки на клієнті (useEffect), весь HTML/JS цих сторінок віддається будь-кому.
- `BikeCard.tsx` — рейтинг хардкоджений як `rating={0}`, тож усі картки скрізь показують 0 зірок незалежно від реальних відгуків.
- `BookingForm.tsx` — немає перевірки перетину дат на клієнті; ключі помилок `dateConflict`, `pastDate`, `endBeforeStart` визначені в перекладах, але ніде не використовуються.

## MEDIUM

**Backend**
- `EmailService` — при помилці Resend API лист верифікації не надсилається, але `register()` все одно репортує успіх.
- `AuthService.register()` — TOCTOU гонка при перевірці унікальності email (рідкісний випадок, але дає 500 замість чистого 400).
- `BikeService.deleteBike()` — не скасовує і не перевіряє активні бронювання при видаленні велосипеда.
- `AdminService.deleteUser()` — soft-delete без каскаду: велосипеди й бронювання видаленого користувача лишаються активними/видимими.
- `CreateBookingRequest.startDate` — анотація `@Future` забороняє бронювання "на сьогодні", хоча документація стверджує, що це дозволено.
- Глобальний `server.error.include-message: always` — підвищує ризик витоку деталей помилок поза кастомним handler-ом.

**Frontend**
- Немає CSRF-токена при cookie-based auth (потрібно перевірити очікування backend).
- API-interceptor втрачає HTTP-статус помилки → 401 і 500 обробляються однаково (сесія може скидатись через тимчасовий збій сервера).
- `admin/users/page.tsx` — debounce-таймер зберігається в `window` замість `useRef`, не очищується при unmount.
- `Navbar.tsx` — перемикач мови робить `pathname.replace()` рядковою заміною, губить query-параметри (наприклад, фільтри пошуку) і потенційно ламається при збігу підрядка.
- Багато хардкоджених рядків англійською/двомовно прямо в JSX замість ключів перекладу — німецька локалізація для цих місць не працює (dashboard, admin/bikes, bike detail тощо).
- `BikeForm.tsx` — немає клієнтської валідації файлів фото (тип/розмір/кількість), хоча ключ перекладу "максимум 5 фото" існує.

## LOW

- `BaseEntity` без `equals`/`hashCode` (теоретичний ризик, не використовується в Set/Map — не критично).
- `ReviewService` — дублюючий check-then-act на унікальність відгуку (DB constraint захищає від реальної шкоди, лише статус помилки не ідеальний).
- `BikeService.rejectBike()` — немає guard-у на повторне відхилення (асиметрично з `approveBike()`).
- Кілька дрібних "мертвих" фрагментів коду (зайвий spread в edit-сторінці велосипеда, застарілий шлях `src/pages` у `tailwind.config.ts`, безглуздий type-cast у `DashboardLayout.tsx`).
- `useAuth.login` робить зайвий повторний запит профілю замість використання даних із відповіді логіну.

## Підсумок
Найкритичніше: можливий double-booking велосипедів, недосяжна функція завантаження фото, ризик підробки JWT при відсутності env-змінної в проді, неузгодженість enum категорій велосипедів між Java і БД. Рекомендую виправляти в такому порядку: критичні → high → medium.
